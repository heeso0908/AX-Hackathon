# VoC 분류 및 판단 기준

> 대상 모듈: `src/classify.py` | 입력: `output/<mode>/voc-cleaned.csv` | 출력: `output/<mode>/voc_classification.csv`

## 1. 분류 목적

CS팀은 빠르게 응대 우선순위를 파악하고, 제품팀은 반복 패턴을 제품 개선 과제로 전환할 수 있도록 설계했다.
불만/기능 요청/칭찬/일반 문의 4유형과 urgency·customer_impact를 함께 산출해, "무엇을 먼저 처리해야 하는가"와
"어떤 기능을 다음에 만들어야 하는가"를 동시에 답할 수 있게 했다.

## 2. 4유형 분류 기준

| 유형 | 판단 신호 | 대표 키워드/패턴 | 근거 |
|------|-----------|------------------|------|
| 불만 | 장애·오류·불일치 등 명확한 부정 신호가 있음 | 장애, 오류, 불편, 마비, 반려, 안 됩니다, 틀린, 누락, 끊김, 제한적, 차이납니다 | 서비스 실패·업무 차질을 알리는 표현은 최우선으로 감지해야 이탈 위험을 조기에 잡을 수 있음 |
| 기능 요청 | 신규 기능/가이드/템플릿 등 "만들어 달라"는 신호가 중심 | 제공해주세요, 추가해주세요, 만들어주시면, 지원해주세요, 요청드립니다, (+ 기능/가이드/템플릿/대시보드 등 대상 명사와 결합된 "좋겠습니다/가능한가요/필요합니다") | 명확한 액션 요청 동사만 강한 신호로 인정하고, "설명해주세요"류 정보성 요청은 일반 문의로 남겨 오탐을 줄임 |
| 칭찬 | 감사·만족 등 긍정 표현이 중심 | 감사합니다, 만족스럽, 좋았습니다, 친절하게, 줄었/줄어들었/올라갔(효율·품질 개선 서술) | 세일즈 레퍼런스·케이스 스터디 후보를 자동으로 걸러내기 위한 신호 |
| 일반 문의 | 위 세 신호가 없고 정보 확인이 중심 | 언제, 어떻게, 방법, 기준, 절차, 가능한가요, 알고 싶다, 궁금합니다 | 나머지 모든 경우의 기본값(default) — 규제 정보 요구 패턴을 놓치지 않기 위해 가장 넓게 잡음 |

**판단 우선순위**: 불만 → 기능 요청 → 칭찬 → 일반 문의 순으로 검사한다. 즉, 불만 신호가 있으면 다른
신호가 섞여 있어도 불만이 최우선이며, 상세 맥락은 `sub_intent`에 남긴다.

## 3. 혼합 케이스 처리 기준

| 혼합 상황 | 처리 결과 | 판단 근거 |
|-----------|-----------|-----------|
| 불만 + 요청 (예: "안 됩니다... 해결해주세요") | `voc_type=불만`, `sub_intent=개선요청` | 불만이 이탈 위험과 직결되므로 유형은 불만으로 두고, 뒤에 붙은 개선 요구는 제품팀이 놓치지 않도록 sub_intent로 별도 태깅 |
| 문의 + 요청 (예: "지원해줄 수 있나요? 방법론이...") | `voc_type=기능 요청`, `sub_intent=가이드요청` 또는 `기능요청` | 질문형이지만 실제로는 미보유 기능/가이드에 대한 요구이므로 기능 요청으로 승격. 가이드·문서·방법론·교육 관련 단어가 있으면 `가이드요청`, 그 외 기능성 요구는 `기능요청` |
| 칭찬 + 개선 제안 (예: "좋았습니다, 다만 ~해주시면") | 긍정 표현 강도(등장 횟수)와 요청 강도를 비교해 더 강한 쪽으로 분류 | 실제 데이터(44건)에는 이 케이스가 없었지만, 새 CSV 재사용을 위해 "표현 강도 비교" 규칙을 남겨둠. 동률이면 칭찬을 우선(고객 이탈보다 레퍼런스 발굴 가치가 명확할 때만 칭찬 판정) |
| id 043 검증 케이스 | `voc_type=불만`, `sub_intent=개선요청` | "기능이 너무 제한적입니다"(불만) + "기능 개선 요청드립니다"(요청)가 함께 있어 위 1번 규칙이 그대로 적용됨. 문제 명세의 필수 검증 항목이라 `run_special_validations()`로 매 실행마다 자동 확인 |

## 4. 우선순위 판단 기준

**urgency (High/Medium/Low)** — 문면에 등장하는 표현으로만 판단하며, High > Medium > Low 순으로 검사한다.

| 등급 | 트리거 표현 |
|------|-------------|
| High | 긴급, 마감, 반려, 업무 마비, 수출 계약, EU 세관, 오류, 안 됩니다 |
| Medium | 비용, 감사법인/내부 감사/외부 감사(감사 문맥), 인증, 검증, 차이, 연동 끊김 |
| Low | 위 표현이 없는 나머지 (단순 정보 확인·일반 가이드 요청) |

> 주의: "감사"는 "감사합니다"(칭찬)와 "내부 감사"(규제 감사)의 동형어이므로, 감사법인/내부 감사/외부 감사처럼
> 구체적인 구절만 Medium 트리거로 인정한다. 같은 이유로 "오류가 많이 줄었습니다"처럼 부정 키워드 뒤에
> "줄었/개선되었/해결되었" 등 해소 문맥이 오면 해당 키워드를 불만·urgency 신호에서 제외한다 (`_contains_any_guarded`).

**customer_impact (High/Medium/Low)** — 고객 세그먼트 가중치 + urgency 가중치 + 반복 업무 가산점의 합으로 산정한다.

| 요소 | 가중치 |
|------|--------|
| customer_type = OEM / 1차 협력사 | 3 |
| customer_type = 2차 협력사 | 2 |
| customer_type = 기타 | 1 |
| urgency = High | 3 |
| urgency = Medium | 2 |
| urgency = Low | 1 |
| 반복 업무 표현(일일이/반복/번거/매번/수작업) 존재 | +1 |

총점 6점 이상 → High, 4~5점 → Medium, 3점 이하 → Low.
이 기준으로 "OEM/1차 협력사 + High urgency"는 항상 High(3+3=6)가 되고, "기타 + 단순 문의"는
항상 Low(1+1=2)가 되어 문제 명세의 두 기준점을 정확히 만족한다. "2차 협력사 + 반복 업무"는
반복 업무 가산점이 더해질 때 Medium(2+1+1=4)까지 올라가도록 설계했다.

## 5. 실무 적용 시 한계와 개선 방향

- 현재는 rule-based라 문맥 오분류 가능성이 있다. 실제로 개발 중 "설명해주세요"(정보 확인)를 "제공해주세요"류
  기능 요청과 혼동하거나, "오류가 줄었습니다"(칭찬)를 "오류"라는 단어만으로 불만으로 잘못 인식하는 오류를
  발견해 별도 가드 로직으로 수정했다. 이런 동형어·반의 문맥 문제는 키워드 기반 접근의 근본적 한계다.
- 실제 운영 시에는 CS 담당자가 매주 오분류 사례를 리뷰하고, `COMPLAINT_KEYWORDS`/`STRONG_REQUEST_REGEX` 등
  키워드 사전을 업데이트하는 피드백 루프를 운영해야 한다.
- 향후에는 규칙 기반 1차 분류 결과를 LLM이 검토(review)하는 2단계 구조나, 신뢰도가 낮은 건(복수 신호가
  동시에 강하게 감지된 건)만 사람이 확인하는 human-in-the-loop 방식으로 확장하는 것을 권장한다.

## 6. 제품 개선 기획안 우선순위 산정 기준 (src/product_plan.py)

**클러스터링**: 불만/기능 요청 VoC를 6개 후보 클러스터로 묶는다. 1차로는 classify.py가 산출한 `topic`
값으로 매칭하고, `topic == "기타"`인 행만 클러스터별 키워드로 보완 매칭한다(이미 topic이 명확한 행을
키워드로 재매칭해 중복 집계하지 않기 위함).

| 클러스터 | topic 매칭 | 구현 난이도(effort_score) 및 근거 |
|---|---|---|
| CBAM 신고·인증서 | CBAM 신고/인증서 | 1점 — 신고 절차 가이드/체크리스트 수준(문서·템플릿) |
| Scope3 협력사 데이터 자동화 | Scope 3/공급망 데이터 | 3점 — 대량 발송·자동 리마인드 파이프라인(신규 모듈) |
| 보고서 커스터마이징 | 보고서/대시보드 | 2점 — 템플릿 편집기(화면/리포트 개선) |
| 계산 로직 투명성·검증 | 계산 로직/데이터 신뢰성, 검증/인증/감사 | 2점 — 계산 근거 화면 + 문서화(화면/리포트 개선) |
| API/ERP 연동 안정성 | 플랫폼 오류/API/권한 | 3점 — 연동 모니터링·재시도 로직(API/복잡한 연동) |
| FAQ/가이드 온보딩 | (topic 무관, `sub_intent`에 "가이드" 포함 여부로 매칭) | 1점 — FAQ 콘텐츠 수준 |

**점수 산식**: `priority_score = frequency_score + impact_score + urgency_score + regulation_score - effort_score`

| 점수 | 기준 |
|---|---|
| frequency_score | 관련 VoC 1건=1점, 2~3건=2점, 4건 이상=3점 |
| impact_score | 클러스터 내 OEM/1차 협력사 포함=3점, 2차 협력사 포함(고가치 고객 없을 때)=2점, 기타 중심=1점 |
| urgency_score | 클러스터 내 urgency High 포함=3점, Medium 포함(High 없을 때)=2점, Low만 있음=1점 |
| regulation_score | regulation_context에 CBAM/CSRD/Scope3/검증·인증 포함=3점, ESG·공시/PCF·LCA 포함=2점, 그 외=1점 |
| effort_score | 위 클러스터 표 참고 (구현 난이도는 데이터로 산출할 수 없어 클러스터 성격에 따라 고정 배정) |

**동점 처리**: priority_score가 같으면 (1) 관련 VoC 건수가 많은 클러스터, (2) regulation_score가 높은
클러스터를 우선한다. 근거: 빈도가 높을수록 실사용자 영향 범위가 넓고, 규제 연관성이 높을수록 시행일이
정해진 외부 압력(CBAM 2026-01 등)에 대응해야 하므로 지연 비용이 크다.

**최종 실행 결과(2026-07-01, data/voc.csv 기준)**: CBAM 신고·인증서(11점, id006/010/023/045)와 계산
로직 투명성·검증(9점, id014/019/027)이 상위 2건으로 선정되었다. API/ERP 연동 안정성(5점)이 가장 낮은
것은 effort_score가 높은(3점) 반면 관련 VoC의 고객 세그먼트가 대부분 "기타"라 impact_score가 낮게
잡혔기 때문이며, 새 CSV에서 OEM/1차 협력사 비중이 늘면 순위가 바뀔 수 있다.

## 7. 구글시트 연동 방식 결정 (Challenge 도전 항목, src/sheets_export.py)

**배경**: `problem.md`의 도전 항목은 "구글시트 자동 기입 연동 (Cowork 커넥터 활용)"을 요구하지만,
이 커넥터는 claude.ai의 Cowork(채팅) 환경에만 존재하며 지금 제출은 Claude Code(터미널) 환경이다.
`CLAUDE.md`는 "별도 API 키 없음"을 명시적 제약으로 두고 있어, 구글 클라우드 API 키를 필수로 요구하는
방식은 이 제약과 충돌한다.

**결정**: 완전한 실시간 연동을 강제하는 대신 2단계로 나눠 구현했다.
1. **기본(항상 실행, 자격증명 불필요)**: 매 실행마다 `sheets_export.tsv`를 생성한다. 구글시트에서
   전체 선택 후 붙여넣기(Ctrl+V)만 하면 그대로 표가 채워지는 탭 구분 텍스트다.
2. **선택(opt-in, 서비스 계정 필요)**: 사용자가 본인의 구글 서비스 계정 자격증명(JSON)과
   spreadsheet ID를 제공하면(`--push-sheets --sheet-id ... --sheets-credentials ...` 또는
   Streamlit "구글시트에 자동 기입" 폼) `gspread`로 실제 시트에 "VoC 분류"·"요약" 두 워크시트를
   지우고 다시 채워 넣는다(자동 기입).

**근거**: (1) CLAUDE.md의 "별도 API 키 없음" 원칙을 지키면서도 기본 실행이 항상 성공해야 한다,
(2) 그럼에도 채점자가 실제 자동 기입 기능을 테스트하고 싶다면 재현 가능한 형태로 켤 수 있어야
한다, (3) gspread/google-auth 미설치·자격증명 누락 시 크래시 대신 `SheetsExportError`로 원인과
해결 방법을 안내해야 한다(예: "pip install gspread google-auth 필요"). 이 세 요구를 동시에 만족하는
방식이 opt-in 구조다.
